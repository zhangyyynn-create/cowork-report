本期导读

本周 AI cowork 的核心信号不是某个单点模型升级，而是 Agent 产品开始争夺“工作入口”。OpenAI 将 ChatGPT Work 放在桌面端和企业连接器里，Anthropic 则把 Claude Cowork 扩展到手机和浏览器，两者都在把 AI 从聊天窗口推向持续工作的执行层。

这类变化对国内 cowork/buddy 产品的意义很直接：用户真正愿意长期使用的不是一个会写答案的助手，而是一个能读取邮件、会议、文件、IM 和业务系统上下文，并在授权边界内持续推进工作的同事。

从产品建设看，本周值得关注的不是 WorkBuddy 自身，而是外部产品正在把入口、连接器、后台任务、权限和审计组合成完整体验。未来差异点会落在谁能更早拿到高频场景、谁能更可信地执行动作、谁能把结果回写到组织流程里。

桌面端、手机端和浏览器正在成为 AI cowork 的三类常驻入口，单一网页聊天入口会越来越弱。
连接 Slack、Gmail、Drive、Calendar、CRM 等系统后，AI 才能从“问答工具”变成“任务执行者”。
异步执行能力会改变用户预期：用户会希望晚上交代任务、早上看到结果，而不是一直盯着对话框。
企业落地的关键不只是模型能力，还包括权限边界、过程可追溯、失败可恢复和人工确认点。

产品动向

桌面工作台

OpenAI 推出 ChatGPT Work：把聊天、Codex 和企业连接器合成工作入口

2026-07-09 · The Verge / OpenAI · 点击查看原文 →

OpenAI 发布 GPT-5.6 的同时推出 ChatGPT Work，产品把 ChatGPT、Codex 和面向日常工作的连接器组合在一起，覆盖 Slack、Gmail、Google Drive、日历和 CRM 等上下文来源。它强调桌面端优先，并会扩展到 Web 和移动端，面向 Pro、Enterprise、Edu、Plus、Business 等用户逐步开放。

入口不再只是聊天框

ChatGPT Work 的关键不是又多了一个模型，而是把对话、代码、文件、邮件和业务系统放进同一个桌面入口。对 cowork 产品来说，桌面端意味着更贴近日常操作路径，也更容易获得持续上下文。

连接器决定任务深度

Slack、Gmail、Drive、日历和 CRM 连接后，AI 可以理解任务背景、相关人、材料和时间约束。没有连接器的 AI 助手只能回答问题，有连接器后才可能形成“读上下文、做判断、产出材料、回写系统”的闭环。

简评：这说明 OpenAI 正在把 ChatGPT 从通用助手推向工作操作系统。对国内 buddy 产品的启发是：不能只做对话和插件列表，而要围绕真实办公流设计入口和权限。例如会议前自动汇总材料、销售跟进前读取 CRM、项目复盘时拉取文档和 IM 脉络，这些才是 cowork 的可留存场景。
跨端常驻 Agent

Claude Cowork 扩展到手机和浏览器：异步任务成为核心卖点

2026-07-07 · WIRED / Anthropic · 点击查看原文 →

Anthropic 将 Claude Cowork agent 扩展到智能手机和浏览器，不再要求用户保持桌面端在线。该能力可以从邮件、Slack、会议记录等上下文中提取信息，并在用户离线或夜间继续处理任务，先面向 Max 计划 beta，后续再扩展到 Pro。

从在线助手到后台同事

一旦 Agent 不再依赖电脑开着，用户就会把它当成异步同事：晚上处理资料、早上交付结果、会议前整理上下文。这会让 cowork 从“临时工具”变成“后台任务层”。

跨端不是同步界面，而是同步责任

手机和浏览器入口的价值不只是多一个端，而是让用户在任何时刻都能布置、查看和接管任务。真正的产品挑战在于任务状态、权限、失败恢复和结果解释要跨端一致。

简评：Claude Cowork 的方向非常值得借鉴：AI cowork 不是把 PC 功能搬到移动端，而是让用户可以在碎片时间启动和验收任务。国内产品如果要做 AI 同事，移动端不应只是聊天窗口，而应有任务队列、执行进度、来源证据和确认按钮。
任务工作负载

ChatGPT Work 覆盖财务、数据分析和工程任务：可执行工作负载成为包装方式

2026-07-11 · TechRadar · 点击查看原文 →

TechRadar 对 ChatGPT Work 的报道强调，它面向财务、数据分析、工程等工作负载，支持信息收集、材料创建、长时间运行流程和插件式工具调用。桌面客户端在授权后可以获得一定的计算机控制能力，用于跨应用执行任务。

按工作负载而不是按功能命名

OpenAI 的表达开始从“我能写邮件、写代码”转向“我能处理财务、数据分析和工程工作”。这意味着 AI 产品会越来越用业务问题包装能力，而不是让用户自己拼功能。

计算机控制带来边界问题

桌面控制能力提高了执行力，也放大了误操作和权限风险。cowork 产品必须让用户清楚知道 AI 能看什么、能点什么、能改什么，并在关键动作前设置确认点。

简评：这条新闻的启发在于包装方式：未来 cowork 产品不应该只说“支持文件、邮件、网页”，而要做成可理解的工作负载，比如“竞品周报生成”“客户跟进准备”“会议后行动项追踪”。用户买的是结果，不是功能清单。
产品对比

ChatGPT Work 与 Claude Cowork 对比升温：行业开始围绕“AI 同事”定义新赛道

2026-07-10 · Google News / Jimo AI · 点击查看原文 →

围绕 ChatGPT Work 与 Claude Cowork 的第三方比较开始出现，说明市场正在把这类产品作为独立赛道讨论。对比维度集中在入口、任务持续性、上下文连接、企业治理和执行权限上，而不是单纯模型参数。

赛道语言正在形成

当媒体开始并列比较 ChatGPT Work 和 Claude Cowork，说明“AI cowork / AI 同事”已经不只是内部概念，而是被外部市场识别为新产品类别。

比较维度会倒逼产品补齐

入口、连接器、异步任务、权限、审计和结果质量会成为默认比较项。缺其中任何一块，产品都会被归类为普通助手，而不是 cowork。

简评：对我们更重要的是借用这套外部比较框架审视自身产品：我们是否有明确入口，是否能持续拿到上下文，是否能跨系统执行，是否能证明过程可信。只有这些问题回答清楚，才有资格进入 AI cowork 赛道。

对 cowork 产品建设的启发

1. 优先做高频入口，而不是堆功能。桌面、IM、浏览器和移动端需要形成任务连续性。

2. 连接器是护城河。邮件、会议、文档、IM、CRM 和知识库接入越深，AI 同事越难被替代。

3. 异步任务需要产品化。任务队列、进度、失败原因、证据链和人工确认必须成为基础组件。

4. 对外表达应围绕工作负载，而不是能力枚举，让用户直接理解 AI 能替他完成哪类工作。
