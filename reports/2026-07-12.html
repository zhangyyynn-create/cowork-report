产品动向

产品发布

OpenAI 发布 ChatGPT Work，跨应用 Agent 正式进入办公场景

2026-07-09 · BNN Bloomberg · BNN Bloomberg

OpenAI 于 7 月 9 日推出 ChatGPT Work，这是一个内置于 ChatGPT 的 Agent 功能，能够跨多个应用和文件执行任务。用户可以通过自然语言指令让 ChatGPT Work 操作日历、邮件、文档、表格等办公工具，实现自动化工作流。该产品目前面向 ChatGPT Enterprise 用户开放，是 OpenAI 在 workplace AI 领域的最新布局，直接与 Microsoft Copilot、Google Gemini for Workspace 等产品竞争。

产品动作

ChatGPT Work 并非独立应用，而是 ChatGPT 内部的一个 Agent 模式。用户可以在对话中激活它，并赋予其跨应用操作权限。它支持与 Google Workspace、Microsoft 365、Slack、Notion 等主流办公工具的集成，能够执行如“整理收件箱并回复紧急邮件”“根据上周销售数据生成周报并发送给团队”等复合任务。OpenAI 强调其遵循用户设定的权限边界，每次跨应用操作前会请求确认。

能力边界

目前 ChatGPT Work 仍处于早期阶段，支持的任务类型以信息聚合、文档编辑、日程管理为主，尚未开放自定义 Agent 或复杂多步骤工作流编排。其执行可靠性依赖于底层模型（GPT-4o 系列）的指令遵循能力和工具调用准确性。在涉及敏感数据或关键业务决策时，用户仍需人工复核。OpenAI 计划后续加入更多企业级功能，如审计日志、管理员策略控制等。

场景落点

ChatGPT Work 瞄准的是知识工作者的日常重复性任务，如邮件处理、会议安排、报告生成等。它降低了使用 AI 完成跨应用工作流的门槛——用户无需编写代码或配置复杂集成，只需自然语言描述即可。对于已经使用 ChatGPT Enterprise 的组织，这是一个自然的增值功能，可能加速企业从“试用聊天机器人”向“部署 AI 同事”的转变。

组织/商业影响

OpenAI 此举直接挑战 Microsoft 和 Google 在办公 AI 领域的地位。Microsoft Copilot 深度绑定 365 生态，Google Gemini 依托 Workspace，而 ChatGPT Work 则试图以“中立集成者”身份连接多个平台。如果 OpenAI 能持续扩大集成范围并保证稳定性，可能吸引那些不希望被单一云厂商锁定的企业客户。不过，其定价尚未公布，若显著高于现有 ChatGPT Enterprise 订阅，可能影响采用率。

风险与限制

跨应用权限管理是最大风险点。ChatGPT Work 需要访问用户的日历、邮件、文档等敏感信息，一旦权限控制不当或模型被诱导执行非授权操作，可能造成数据泄露或业务中断。此外，目前仅支持预设的办公工具，对垂直行业软件（如 SAP、Salesforce）的集成尚未提及，限制了其在专业领域的适用性。

PM 简评：ChatGPT Work 的发布是 AI cowork 产品走向主流的关键节点。它证明了“通用 Agent 工作流”不再是概念，而是可交付的产品。对于竞品而言，OpenAI 的品牌影响力和现有用户基础是巨大优势；但产品成熟度、集成深度和安全性仍需时间验证。PM 应关注其后续的权限模型设计、任务失败率以及企业客户的真实反馈——这些将决定 ChatGPT Work 是成为“AI 同事”的标杆，还是又一个被束之高阁的演示工具。
产品升级

Akeneo 发布 Agentic PXM 平台，多 Agent 协作管理产品信息

2026-07-08 · TMCnet · TMCnet

Akeneo 在 7 月 8 日的 Summer Release 中，将其产品云（Product Cloud）升级为“完全 Agentic 平台”。新平台引入多 Agent 协作机制，不同 Agent 负责产品信息管理（PXM）的不同环节，如数据采集、质量校验、分类映射、渠道分发等。这些 Agent 可以自主协调任务，并在需要时与人类协作。Akeneo 称这是从“记录系统”到“行动系统”的转变，旨在减少人工操作，加速产品上市。

产品动作

Akeneo 的 Agentic 平台并非单一 Agent，而是一组专业 Agent 组成的“舰队”。例如，一个 Agent 负责从供应商处抓取产品数据，另一个 Agent 校验数据完整性，第三个 Agent 根据规则自动分类并推送到电商渠道。用户可以通过仪表盘监控 Agent 工作状态，并在异常时介入。Akeneo 强调所有 Agent 操作均可追溯，并保留人工审批节点。

能力边界

该平台目前聚焦于产品信息管理这一垂直场景，Agent 的能力边界由预定义的业务规则和模型训练数据决定。对于非标准化的产品数据（如手工品、定制化商品），Agent 可能需要更多人工干预。Akeneo 计划后续开放 Agent 自定义能力，允许用户根据自身业务流程训练或配置 Agent。

场景落点

对于拥有大量 SKU 和多渠道销售的零售/制造企业，产品信息管理是典型的高重复、低创造性的工作。Akeneo 的 Agentic 平台直接切入这一痛点，将人力从数据搬运中解放出来，转向策略性工作。这是 AI cowork 在垂直领域落地的典型案例——不是取代人类，而是让人类专注于更高价值的决策。

组织/商业影响

Akeneo 的升级表明，垂直 SaaS 厂商正在将 Agent 能力作为差异化竞争点。对于企业客户，这意味着他们可以在不更换核心系统的情况下获得 AI 同事能力，降低了迁移成本。但这也给传统 PIM 厂商带来压力，迫使他们加速 AI 化。Akeneo 的 Agent 舰队模式可能成为其他垂直 SaaS 的参考模板。

风险与限制

多 Agent 协作的可靠性是关键挑战。如果某个 Agent 产生错误数据，可能被下游 Agent 放大，导致系统性偏差。Akeneo 需要提供强大的监控和回滚机制。此外，Agent 的决策透明度仍需提升——用户需要理解 Agent 为何做出某个分类或分发决策，否则难以信任。

PM 简评：Akeneo 的 Agentic 平台是本周最值得关注的垂直领域 AI cowork 案例。它展示了如何将 Agent 工作流嵌入现有业务系统，而不是要求用户迁移到全新的“AI 优先”平台。对于产品经理而言，Akeneo 的“专业 Agent 舰队”模式比通用 Agent 更易落地，因为其任务边界清晰、失败影响可控。但这也意味着通用型 AI cowork 产品（如 ChatGPT Work）需要更深入地与垂直软件集成，才能在企业核心流程中占据一席之地。
弱信号

Constructor 在 Gartner 关键能力报告中获多项第一，但非直接 cowork 产品

2026-07-09 · TMCnet · TMCnet

Gartner 发布《搜索与产品发现关键能力报告》，Constructor 在“GenAI 产品发现”“产品搜索与发现”“多站点与多渠道”三个用例中排名第一。Constructor 是一家专注于电商搜索和产品发现的 AI 公司，其平台利用大模型提升搜索结果的相关性和个性化。该报告评估了 12 家供应商，Constructor 在多个维度表现突出。

能力边界

Constructor 的核心能力是 AI 驱动的产品搜索和推荐，而非通用办公任务执行。它属于 AI 增强型工具，而非 AI cowork 产品。其 Agent 特性体现在自动优化搜索规则、动态调整排序等方面，但用户交互仍以搜索框和结果页为主，缺乏跨应用工作流执行能力。

场景落点

对于电商企业，Constructor 可以显著提升产品发现效率，间接减少运营人员的手动调优工作。但它并不直接替代人类完成“工作”，而是优化人类的工作结果。因此，它更接近 AI 辅助工具，而非 AI 同事。

对 cowork 产品的启发

Constructor 的成功提醒我们，AI cowork 产品不应只关注“执行任务”，还应关注“增强决策”。在搜索和推荐场景中，AI 可以成为用户的“智能副驾”，提供建议而非直接操作。这种模式在合规要求高的行业（如金融、医疗）可能更受欢迎。

PM 简评：Constructor 的 Gartner 排名是本周的弱信号，因为它并非严格意义上的 AI cowork 产品。但它展示了 AI 在垂直场景中“增强而非替代”的价值。对于 AI cowork 产品经理，这意味着在追求“全自动执行”的同时，不应忽视“智能辅助”这一更易落地的路径。尤其是在企业客户对 Agent 自主性仍有顾虑的当下，提供可解释、可干预的 AI 建议可能比全自动 Agent 更容易获得信任。

关键判断

1. OpenAI 推出 ChatGPT Work，通用型 AI cowork 产品进入巨头正面竞争，产品形态从聊天助手升级为跨应用 Agent。

2. Akeneo 的 Agentic PXM 平台展示了垂直领域多 Agent 协作的落地路径，专业 Agent 舰队模式比通用 Agent 更易嵌入现有业务。

3. ChatGPT Work 的跨应用权限管理是最大风险点，其安全性设计将影响企业采用意愿。

4. 垂直 SaaS 厂商加速 AI 化，可能成为 AI cowork 产品的重要渠道，通用型产品需深化与垂直软件的集成。

5. Constructor 的 Gartner 排名提示：AI 增强（辅助决策）与 AI 替代（自动执行）是两条并行路径，产品经理应根据场景选择侧重。
