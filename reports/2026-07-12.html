本期导读

本周 AI cowork 的主线不是单个模型参数升级，而是两类产品形态同时变清晰：OpenAI 通过 ChatGPT Work 把桌面端、Codex 与企业连接器组合成工作入口；Anthropic 则把 Claude Cowork 推向手机和浏览器，让 Agent 可以脱离桌面端持续执行。

这背后的竞争逻辑很像办公软件平台化的早期阶段：谁能更自然地进入用户的日常工作流，谁就更容易拿到邮件、会议、文件、IM 和业务系统上下文；谁能把任务状态、执行证据和人工确认做清楚，谁才可能从“工具”变成“同事”。

对国内 AI cowork / buddy 产品来说，本周外部动态最值得看的不是功能清单，而是产品组织方式：入口必须高频，连接器必须足够深，异步任务必须可追踪，执行动作必须有权限和审计。缺其中任何一块，都很难形成长期使用习惯。

AI cowork 的竞争焦点正在从“模型能不能回答”转向“产品能不能持续接入工作上下文并完成任务”。
桌面端、手机端和浏览器都在变成常驻入口，单一网页聊天框会越来越像临时工具。
连接器决定任务深度：没有邮件、会议、文档、IM、CRM 等上下文，Agent 很难承担真实工作。
异步执行会抬高用户预期：用户会希望交代一次任务后，在不同端查看进度、证据和结果。

产品动向

桌面工作台

OpenAI 推出 ChatGPT Work：把聊天、Codex 和企业连接器合成工作入口

2026-07-09 · The Verge / OpenAI · 点击查看原文 →

OpenAI 在发布 GPT-5.6 的同时推出 ChatGPT Work。它不是单独增加一个聊天入口，而是把 ChatGPT、Codex、桌面端和企业连接器放在同一套工作体验里：Slack、Gmail、Google Drive、Calendar、CRM 等外部上下文被纳入任务处理链路，桌面端先行，再逐步覆盖 Web 和移动端。

从“聊天产品”向“工作操作层”迁移

ChatGPT Work 的关键变化，是把 AI 放进工作发生的地方，而不是等用户把上下文复制到聊天框。桌面端意味着它更接近文件、浏览器、会议和开发环境；Codex 的并入则说明 OpenAI 想把“写代码”扩展成更广义的工作执行能力。

连接器是进入企业场景的门票

Slack、Gmail、Drive、日历和 CRM 连接后，AI 可以知道任务来自谁、关联材料在哪里、时间约束是什么、结果要交付给哪个系统。这个链路一旦打通，AI 就不再只是生成文本，而是在组织信息流里承担具体环节。

对国内产品的直接对标点

飞书、钉钉、企业微信等办公入口天然拥有组织关系、消息、会议和文档上下文，但如果 AI 只停留在问答助手层，优势会被削弱。真正值得做的是把 AI 嵌到“会前准备、会后跟进、项目复盘、客户推进”等高频闭环里。

简评：简评：OpenAI 这一步本质上是在抢“默认工作台”，不是在做一个更聪明的聊天窗口。国内产品的机会在于办公入口和组织数据更近，但风险也在这里：如果只把 AI 做成侧边栏，用户很快会把它当成可替换插件；如果能把任务触发、材料读取、结果回写和权限确认串起来，才有可能形成真正的 cowork 粘性。
跨端常驻 Agent

Claude Cowork 扩展到手机和浏览器：异步任务成为核心卖点

2026-07-07 · WIRED / Anthropic · 点击查看原文 →

Anthropic 将 Claude Cowork agent 扩展到手机和浏览器，不再要求用户保持桌面端在线。据报道，它可以读取邮件、Slack、会议记录等上下文，并在用户离线或夜间继续处理任务。产品先面向 Max 计划 beta，再扩展到 Pro，方向很明确：让 Agent 从在线问答变成可托付的后台工作者。

跨端不是多做一个界面，而是让任务不断线

手机和浏览器入口的价值，不只是让用户随时聊天，而是让用户随时布置、检查和接管任务。一个 AI 同事如果只能在电脑前使用，就更像工具；如果能跨端保留任务状态，它才开始接近真实协作。

异步执行会重塑产品交互

当用户可以在晚上交代任务、早上查看结果，产品界面就必须从“对话流”升级为“任务流”：状态、步骤、证据、失败原因、待确认动作都要可见。否则 Agent 越能执行，用户越不敢把重要任务交给它。

移动端应承担验收和接管，而不只是输入

移动端最适合处理轻量决策：确认下一步、查看证据、驳回错误结果、补充上下文。对于 AI cowork 产品，移动端的重点不是复制桌面端能力，而是把任务队列和人工确认做得足够顺手。

简评：简评：Claude Cowork 的价值在于把“AI 帮我做”从即时对话拉到异步协作。这个方向会逼迫所有 cowork 产品补齐任务管理能力：不只是让 AI 会做事，还要让用户看得懂它怎么做、做到了哪一步、哪里需要人接管。国内产品如果移动端只有聊天窗口，会错过最关键的验收和调度场景。
工作负载包装

ChatGPT Work 覆盖财务、数据分析和工程任务：可执行工作负载成为新包装方式

2026-07-11 · TechRadar · 点击查看原文 →

TechRadar 对 ChatGPT Work 的报道强调，它面向财务、数据分析、工程等工作负载，支持信息收集、材料创建、长时间运行流程和插件式工具调用。桌面客户端在授权后可以获得一定的计算机控制能力，用于跨应用执行任务，这让产品从“生成内容”进一步走向“完成工作”。

产品话术从功能清单转向任务结果

“支持文件、邮件、网页、代码”是能力描述，“处理财务分析、数据整理、工程任务”才是用户能理解的工作负载。OpenAI 的包装方式说明，AI cowork 要降低理解成本，最好按具体工作场景交付。

计算机控制能力是效率杠杆，也是风险入口

一旦 AI 可以跨应用点击、读取和修改内容，执行力会明显提升，但误操作、越权访问和责任归属也会同时放大。企业用户不会只问能不能做，还会问每一步有没有记录、能不能预览、关键动作能不能暂停。

场景模板会变成产品资产

如果一个产品能把“竞品周报生成”“客户拜访准备”“投研材料初筛”“研发需求拆解”做成可复用模板，就比通用提示词更容易沉淀用户习惯。模板背后的流程、连接器和确认点会成为壁垒。

简评：简评：这条新闻最值得借鉴的是“工作负载化”表达。AI cowork 不应只展示自己能调用多少工具，而要告诉用户：哪类工作可以交给它、需要哪些授权、输出会回到哪里。机会在于高频场景模板，限制在于执行边界必须清楚；否则越自动化，企业越谨慎。
赛道对比

ChatGPT Work 与 Claude Cowork 被并列讨论：行业开始围绕“AI 同事”定义新赛道

2026-07-10 · Google News / Jimo AI · 点击查看原文 →

围绕 ChatGPT Work 与 Claude Cowork 的第三方比较开始出现，说明市场正在把这类产品作为独立赛道讨论。对比维度集中在入口、任务持续性、上下文连接、企业治理和执行权限，而不是单纯模型参数或聊天体验。

赛道语言正在形成

当媒体把 ChatGPT Work 和 Claude Cowork 放在同一张比较表里，说明“AI cowork / AI 同事”正在从产品宣传语变成可比较的类别。类别一旦形成，用户和企业采购就会用稳定维度筛选产品。

比较维度会倒逼补齐短板

入口、连接器、异步任务、权限、审计和结果质量会成为默认检查项。缺其中任何一块，产品都会被看成普通助手，而不是能承担工作的 cowork。

国内产品需要建立自己的对标表

不能只看对手发了什么功能，而要把对手映射到能力坐标：入口在哪里、能读什么、能写回哪里、关键动作如何确认、失败如何恢复。这样的对标才有助于产品路线。

简评：简评：赛道成形是一件好事，也会让同质化产品更快出局。未来用户不会满足于“AI 助手”这个宽泛标签，而会追问它到底能不能作为同事参与工作。对产品团队来说，最重要的是尽早建立能力坐标，而不是跟着每条新闻补按钮。

对 cowork 产品建设的启发

1. 先做高频入口，再做能力堆叠。桌面、IM、浏览器和移动端要形成同一套任务状态，而不是各自孤立。

2. 连接器要服务任务闭环。邮件、会议、文档、IM、CRM 和知识库接入后，必须能转化为动作和结果，而不只是检索资料。

3. 异步任务需要独立产品化。队列、进度、证据链、失败原因、人工确认点，是 AI cowork 的基础设施。

4. 对外表达应围绕工作负载：让用户理解 AI 能完成哪类工作、需要哪些授权、结果会交付到哪里。
