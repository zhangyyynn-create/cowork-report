本期导读

本周最强信号来自 Kimi K3：它不是单纯的新模型发布，而是同时落到 Kimi、Kimi Work、Kimi Code 和 API，直接影响长程编码、知识工作、可视化报告和多 Agent 工作流。它引起热议的原因不只是参数规模，而是它把“国产模型是否能支撑复杂 cowork 产品”这个问题从口号变成了可测试的产品事件。

第二条主线是工程化 cowork 的成熟：Qoder Cloud Agents 开始补 Forward Mode、调试器、可观测、Memory Store 读写和预算展示；Qoder IDE 与 Cursor in Slack 则把计划、协作标注、多仓库、跨频道状态更新做进日常研发入口。也就是说，Agent 不再只是生成答案，而是在被包装成能被团队分派、监督、复盘和接力的工作单元。

第三条主线是办公与交付场景继续被平台吞进去：TRAE Work 把官方知识库、真实场景指南和移动端任务管理补上；Manus 把 PPTX 和 Auto-Publish 做成最终交付链；OpenAI、Anthropic、Google 则分别从桌面 Work、教育场景、Gmail/Docs 高频入口推进。cowork 产品的竞争正在从“谁更像助手”走向“谁更像一个能交付、可审查、可长期使用的工作系统”。

国产 Agent 底座

Kimi K3 发布并接入 Kimi Work / Kimi Code：长程 Agent 能力被推到前台

2026-07-16 / 2026-07-17 · Moonshot AI 官方 / AP / Tom's Hardware / MarketWatch · Moonshot Kimi K3 官方博客 / Moonshot 官网 / AP 报道 / Tom's Hardware / MarketWatch

Moonshot AI 发布 Kimi K3，官方称其为 2.8T 参数、原生多模态、1M token 上下文的开放 3T 级模型，并明确上线到 Kimi.com、Kimi Work、Kimi Code 和 Kimi API。官方博客把它定位在长程编码、知识工作、推理和多模态创作；AP、MarketWatch、Tom's Hardware 等媒体也围绕其前端编码表现、开源模型竞争和中美 AI 能力差距展开报道。

为什么不是普通模型新闻

K3 的关键在于它不是只作为 API 模型存在，而是同步进入 Kimi Work 和 Kimi Code。对 cowork 产品来说，模型能力只有落到任务入口、工具调用、文件处理和交付形态里，才会改变用户体验。1M 上下文、视觉能力和长程编码能力，意味着它可以承接更大的代码库、长报告、复杂研究链和多轮自我改进任务。

Kimi Work 的产品含义

官方博客展示了 Kimi Work 里的 Widgets、Dashboard、交互式研究报告、可视化图表和长链路研究案例。这说明 Kimi 的工作台不是停在问答助手，而是在尝试把“研究、分析、制图、展示、持续更新”打包成一个工作输出环境。对知识工作 cowork 来说，最终交付不再只是文字，而是仪表盘、网页、PPT、表格和可交互组件。

争议本身也是信号

媒体讨论集中在开放模型冲击、能力接近闭源前沿模型、成本压力以及蒸馏争议。对企业采用者而言，这代表两件事：一方面国产模型底座可能降低 cowork 产品的推理成本和供应链依赖；另一方面，模型来源、许可证、权重发布时间和合规风险会成为采购与部署前必须核查的内容。

对 cowork 产品的启发

K3 让 cowork 产品的能力门槛明显上移：长上下文、视觉理解、代码执行、深度研究、交互式产物会逐渐从“亮点功能”变成基础期望。真正的差异不只是接入哪个模型，而是能否把模型转化成稳定的工作流、明确的任务状态、可编辑产物和可控的企业权限。

PM 简评：Kimi K3 这周值得放在头条，因为它把国产 cowork 产品最核心的底座问题摆到台面：如果本土模型已经能支撑长程编码、复杂研究和多模态产物，国内 cowork 厂商就不必完全围绕海外模型能力节奏做产品。但这不是简单的“有了强模型就赢了”。K3 同时暴露了另一个判断：模型发布会制造热度，产品留存靠入口、状态、文件、交付、协作和治理。Kimi Work 后续是否能把 Widgets/Dashboard 变成真实高频使用，而不是演示级案例，将决定它在 cowork 赛道里是模型光环，还是完整工作台。
研发 Agent

Kimi Code 一周密集迭代：子 Agent、后台任务、计划模式和文档问答补齐长任务体验

2026-07-14 至 2026-07-17 · Kimi Code 官方文档 · Kimi Code What's New / Kimi Code CLI changelog

Kimi Code 在 7 月 14-17 日连续更新：Web UI 支持导出 session，超时的前台 Bash 命令可自动转后台；内置文档问答 skill 可引用官方文档回答产品问题；coder sub-agent 支持后台任务、todo list、plan mode、skill 调用和嵌套 Agent；随后又补充复制最近回复、模型列表刷新和网络错误诊断。

从对话工具变成任务系统

这些更新组合起来看，Kimi Code 正在把 Agent 工作流拆成可管理对象：有 session，可导出；有计划，可预览；有后台任务，可挂起继续；有 todo list，可追踪；有子 Agent，可分工。这比单次代码生成更接近真实研发协作，因为复杂需求往往需要多个阶段、多次检查和中途调整。

后台化是 cowork 的关键能力

长命令超时后不再被杀死，而是转入后台并在完成后回传结果，这个细节很像真实同事的工作方式：任务不应该因为前台窗口切换就中断。对所有 cowork 产品而言，后台任务、状态同步和失败续跑都会成为高频刚需，尤其是生成报告、跑测试、拉数据、执行脚本这类等待时间长的任务。

自带文档问答降低学习成本

内置 /check-kimi-code-docs 让用户在终端里直接查官方文档并获得来源链接，这实际是在把产品知识库接进工作流。未来 cowork 产品不只要理解用户文件，也要理解自己的功能边界和操作说明，否则用户会在“我知道它可能能做，但不知道怎么让它做”这一层流失。

对 cowork 产品的启发

研发 Agent 的迭代正在给通用 cowork 打样：计划、后台、子任务、文档自查、错误诊断、会话导出，都是把 AI 从“聪明回答者”变成“可托付执行者”的必要零件。后续办公 cowork 可以把同样结构迁移到会议、邮件、表格、报告和项目管理。

PM 简评：Kimi Code 的价值不只是跟上 Claude Code / Cursor，而是它用很密集的节奏补 Agent 工程化。这里最值得学的是产品层对不确定性的处理：长任务会超时，所以要后台；用户会迷路，所以要文档问答；执行会失败，所以要诊断；任务会复杂，所以要计划和子 Agent。真正成熟的 cowork 产品不是假装 AI 永远一次成功，而是把失败、等待、切换、复盘都设计进流程。
企业 Agent 平台

阿里 Qoder Cloud Agents 0.5.0：Forward Mode、可观测与 Memory Store 指向企业级交付

2026-07-15 · Alibaba Cloud 文档 · Cloud Agents CN release notes

阿里云 Qoder CN Series 的 Cloud Agents CN 在 7 月 15 日更新 0.5.0：控制台支持 Forward Mode 切换，新增 Forward 模板调试器，可创建调试 session、发送消息、上传附件、查看事件流并处理工具审批；新增可观测总览、用量和额度展示；Memory Store 支持 Agent 主动读写；Agent 也支持原生图片识别。

从个人 Agent 变成可交付模板

Forward Mode 的本质是把 Agent 从个人会话转成可被交付给终端用户的模板。管理员预置 template、identity、工具、技能、文件、记忆和环境变量，再由不同用户调用。这个方向更像企业里的“数字岗位”，而不是一个所有人共用的通用聊天框。

调试器与事件流是企业采用前提

企业要把 Agent 放进真实业务流程，必须能调试、观察和审批。Forward 模板调试器、事件流、工具审批、成功率、token/credit 消耗和 7 日趋势，都是让产品经理、管理员和开发者判断“它到底怎么跑、哪里失败、成本多少”的基础设施。

Memory Store 读写改变上下文策略

当 Agent 可以主动整理和维护记忆，产品就从“每次重新喂上下文”转向“跨任务积累组织经验”。这对 cowork 产品很重要：一个真正像同事的 AI，应该记得角色、偏好、流程、历史输出和例外规则，但也必须提供反腐化、权限隔离和可删除机制。

对 cowork 产品的启发

Qoder Cloud Agents 展现的是企业后台能力：模板化、身份隔离、审批、可观测、额度、记忆、图片理解。它提醒 cowork 产品不能只做前台体验，后端治理能力会直接决定是否能进入公司流程，尤其是涉及 IM、云资源、文件和审批的场景。

PM 简评：阿里的这条动向比普通 IDE 功能更值得关注，因为它把 Agent 产品推进到了企业部署层。很多 cowork 产品容易在 demo 里展示“能完成任务”，但真实落地时卡在身份、权限、成本、调试、审计和渠道分发。Qoder Cloud Agents 的更新说明平台方开始把这些脏活累活产品化。对 cowork 产品来说，前台越像同事，后台越要像一套可治理的业务系统。
研发协作

Qoder IDE 1.14 与团队案例：Canvas 标注、轮次限制和治理化研发成为重点

2026-07-14 / 2026-07-15 · Qoder Changelog / Qoder Blog · Qoder changelog / Qoder Amap case

Qoder 1.14.0 在 7 月 15 日新增 Canvas 分享与标注、/gen-image 文生图、Goal 最大轮次限制和 Chat Feed 内容引用；同周 Qoder 发布高德团队实践文章，强调从 Vibe Coding 走向团队级 AI 开发、Harness Governance 和 SDD。两条信息共同指向：研发 Agent 的价值正在从个人提效转向团队交付与治理。

Canvas 把 AI 输出变成评审对象

Canvas 分享和标注让 Agent 输出不再困在个人对话里，而是可以被同事批注、讨论和迭代。对研发 cowork 来说，代码方案、技术设计、页面草图、测试计划都需要进入评审流。只要 AI 产物要被团队采纳，就必须有协作容器。

轮次限制是自治边界

Goal 最大轮次限制看似是一个小设置，但它是在把 Agent 自治边界显性化。团队不可能接受一个无限尝试、无限消耗、没有停止条件的自动开发者。轮次、预算、权限和回滚会成为企业 Agent 的四条基本护栏。

从 vibe 到 governance

高德案例里“告别 Vibe Coding”的主题很有代表性：AI 代码生成率提高后，问题不再是会不会写代码，而是如何保证需求、设计、实现、测试、评审和上线过程可控。cowork 产品的下一阶段竞争会落在工程治理、流程嵌入和组织协作，而不是生成速度本身。

对 cowork 产品的启发

不管是研发还是办公场景，AI 产物进入组织就需要评审、归档、责任和复用。Canvas、标注、Goal 限制、上下文引用这些能力可以迁移到方案评审、报告审稿、投研结论确认、客户邮件审批等更广泛的 cowork 工作流。

PM 简评：Qoder 本周的价值在于它开始回答“团队怎么用 Agent”这个问题。个人助手的路径是越快越好，团队 cowork 的路径是越可控越好。AI 写得越多，人的瓶颈越会转移到评审、采纳和责任边界上。未来 cowork 产品如果不能把 AI 产物放进团队协作与治理链条，就很容易停留在个人效率工具。

TRAE Work 官方知识库与 App 更新：从功能演示走向场景化上手

2026-07-14 / 2026-07-17 · TRAE 官方更新日志 / 官方社区 · TRAE 更新日志 / TRAE Work 官方社区 / Introducing TRAE Work

TRAE 官方更新日志显示，7 月 17 日 TRAE Work 发布官方知识库，覆盖 7 大真实工作场景、30+ 实战指南和 40w+ 字场景应用；7 月 14 日 TRAE App 支持对话流图片预览、HTML 格式产物分享、任务搜索与 Pin 置顶、站内信通知。官方社区同周也出现大量围绕 Work 使用、项目记忆、Design/Code 协作和新闻监控系统的实践讨论。

知识库是降低上手门槛的产品动作

TRAE Work 从 SOLO 升级成面向多角色的 AI 工作助手后，真正的难点不是告诉用户“它能做很多事”，而是让用户知道具体怎么用。官方知识库把真实场景和实战指南体系化，说明字节在补场景教育和方法论，而不是只靠功能发布拉新。

移动端与 HTML 分享强化交付链

App 侧支持图片预览、历史任务搜索、Pin 置顶和 HTML 产物分享，都是为了让任务结果更容易查找、展示和传播。cowork 产品如果要进入办公流程，不能只在生成时好用，还要在后续复用、汇报、查看进度和分享成果时顺手。

社区热度暴露真实使用场景

官方社区里出现项目记忆、Design 模式、Work+Code 接力、新闻动态监控等讨论，说明用户正在把 TRAE Work 用到非纯研发任务上。这些内容不能等同于官方能力发布，但可以作为产品定位正在被用户理解和迁移的信号。

对 cowork 产品的启发

字节这条线的启发是：cowork 产品需要把功能说明升级成任务方法论。越是面向普通办公用户，越不能只给 Prompt 示例，而要给“场景-步骤-产物-复盘”的完整路径。知识库、模板库、示例项目和可分享产物，会成为产品冷启动的重要抓手。

PM 简评：TRAE Work 本周不是最炸裂的模型新闻，但对 cowork 产品很实用。很多产品把能力堆出来后，用户仍不知道该怎么开始；TRAE 选择用场景知识库和社区实践把使用路径铺出来。这说明 buddy/cowork 的竞争不只是 Agent 能力，还有用户教育和产物传播。谁能把复杂任务变成可照着做的工作方法，谁就更容易从尝鲜走向持续使用。

Cursor in Slack 升级：计划先行、多仓库环境和跨频道工作把 Agent 带进团队消息流

2026-07-17 · Cursor Changelog · Cursor changelog

Cursor 在 7 月 17 日更新 Slack 集成：Agent 开始前会先给出计划，执行中持续更新状态；支持命名多仓库环境，遇到需要切换仓库时会提示用户确认；同时支持跨 Slack 频道和线程读取上下文、发送消息，并优化表格、PR 和工件的渲染。

Slack 正在变成 Agent 调度台

Cursor 这次更新不是简单做一个 Slack bot，而是让团队可以在已有沟通流里分派研发任务、观察进度、跨频道收集上下文，并把结果回传到讨论现场。对 cowork 产品而言，高频入口比独立 App 更容易形成习惯。

计划先行降低黑箱感

Agent 先给计划再开始执行，是高自治产品很关键的信任设计。用户可以在早期介入和调整方向，而不是等 Agent 跑完才发现路径错了。状态更新则让长任务从黑箱等待变成可跟踪流程。

多仓库支持更接近真实研发组织

企业研发很少只有一个仓库。前端、后端、数据、SDK、自动化测试通常分散在多个 repo。Cursor 支持命名多仓库环境，说明 Agent 开始进入真实组织结构，而不是只处理个人机器上的单项目。

对 cowork 产品的启发

企微、飞书、Slack、Teams 等消息入口会成为 cowork 产品的关键战场。真正有价值的不是在群里回答问题，而是能围绕团队上下文启动任务、给计划、要确认、发进度、交付可点击结果，并把过程沉淀回项目。

PM 简评：Cursor 的更新说明研发 Agent 正在成为团队流程的一部分。它值得关注的不是“Slack 里也能用 Cursor”，而是它把计划、状态、多仓库和跨频道上下文这些协作要素拉到了消息入口。对通用 cowork 产品来说，这是一条很清楚的路线：不要逼用户去新工作台，先嵌入他们已经在用的沟通场，再把任务执行做深。
交付闭环

Manus 一周两次产品更新：Auto-Publish 与原生 PPTX 把 Agent 产物推向可交付

2026-07-13 / 2026-07-14 · Manus Blog · Manus Auto-Publish / Manus PowerPoint Slides / Ascendea customer story

Manus 在 7 月 13 日发布 Auto-Publish，用户开启后，网站构建成功即可自动部署到公开 URL；7 月 14 日发布 PowerPoint Slides，支持直接生成原生 .pptx，包含可编辑图表、表格和视觉内容。同周客户案例还把 Manus 描述为公司里的 AI chief of staff，用于复杂研究、SOP、技术方案和自动化。

自动发布缩短最后一公里

很多 Agent 能生成网页或应用，但最后仍需要人工打开发布菜单、确认设置、点击上线。Auto-Publish 把这个步骤变成开关，意味着 Manus 在追求“从想法到上线”的连续闭环。它的边界也很清楚：默认关闭，只有用户决定持续发布时才开启。

原生 PPTX 解决商务交付痛点

PowerPoint 模式不是把网页截图导出，而是生成结构化 .pptx，图表和表格可以继续编辑。这对商务、咨询、投研和内部汇报很重要：AI 产物如果不能在现有办公软件里二次修改，就很难进入正式交付链。

从生成到运营

Manus 的这两项更新分别补“上线”和“文档交付”，再加上客户案例里把 Manus 放在复杂任务顶层的组织方式，说明其产品方向是让 Agent 不止做内容，而是连接研究、制作、发布、迭代和运营。

对 cowork 产品的启发

cowork 产品不能满足于产出一个看起来不错的回答，必须关心结果能否发布、能否下载、能否编辑、能否被同事接着用。发布、导出、版本、审批和回滚会决定 Agent 是否真的进入工作交付链。

PM 简评：Manus 本周的两个动作都很产品经理：一个解决“做完怎么上线”，一个解决“交付物能不能进入 PowerPoint”。这比继续堆更多生成能力更实际。对 cowork 产品来说，用户评价的是任务有没有完成，而不是模型有没有回答。能把产物送到公开网站、PPT、表格、邮件或业务系统里的产品，会比只在聊天框里展示结果的产品更接近真实同事。

ChatGPT 桌面 Work 与搜索增强：项目、文件、跨设备继续工作成为通用 Agent 底座

OpenAI 在 7 月 14 日增强 ChatGPT 对聊天、项目、图片、文档和文件的统一搜索；7 月 16 日更新桌面端体验，增加 Chat / Work 切换、统一 Recents、桌面端 Projects、Cloud Work 跨 web、mobile、desktop 继续工作。其 Slack agents 文档也显示企业用户可在 Slack 中调用 Agent、连接系统、输出文件并设置定时消息或结果推送。

Work 成为桌面里的独立工作模式

Chat / Work 的区分说明 OpenAI 正在把“问答”和“做事”拆成两种心智。Chat 负责快速咨询，Work 负责长任务、文件、项目和跨设备延续。对 cowork 产品来说，这种信息架构会影响用户预期：复杂任务应该有自己的历史、状态和上下文，而不是混在聊天记录里。

统一搜索是记忆入口

跨聊天、项目、图片、文档和文件搜索让 Agent 更容易找回过去上下文。持续 cowork 的前提是它能知道用户以前做过什么、文件在哪、项目进展如何。没有统一检索，AI 就只能像临时外包；有了可搜索记忆，才可能像长期同事。

Slack agents 补团队分发

Slack agents 强调频道响应、外部系统连接、输出文件和定时结果，这把 ChatGPT 从个人工具推进到团队工作流。它与 Cursor in Slack 的方向相似：Agent 要待在团队沟通现场，而不是只存在于独立应用里。

对 cowork 产品的启发

通用 cowork 的核心链条会是：个人/项目记忆、连接器、跨设备任务状态、团队消息入口、定时交付和审批确认。OpenAI 的优势是入口和生态；其他产品要差异化，就要在行业场景、交付格式或企业治理上做得更深。

PM 简评：OpenAI 本周的动作不如 Kimi K3 热闹，但更像长期平台建设。它把 Work 放进桌面、把搜索打通、把 Slack agents 做成团队入口，这些都在强化“ChatGPT 是工作层”的定位。对 cowork 产品来说，OpenAI 的威胁不是某个功能，而是它正在把入口、上下文和交付连成默认路径。独立产品要避开被平台内置能力吞掉，就必须找到更具体、更深的岗位或行业工作流。

Anthropic Claude for Teachers：把技能、资料库、定时任务和隐私条款打包进教师工作流

2026-07-14 · Anthropic 官方 · Anthropic Claude for Teachers / Claude Cowork scheduled tasks

Anthropic 发布 Claude for Teachers，面向美国 K-12 教师提供免费 premium Claude 能力、教学 skills、Learning Commons 资料连接，并映射到 50 个州的学术标准。官方示例包括分析班级数据、按学生情况调整教学，以及每天 4 点自动复盘 exit tickets 并适配第二天教学计划。

垂直化不是换皮

Claude for Teachers 的重点不是把聊天助手改名给老师，而是把教学资源、课程标准、课堂数据、例行复盘和隐私条款一起打包。垂直 cowork 的价值来自“默认知道这个岗位怎么工作”，而不是让用户从空白提示词开始。

定时任务让 AI 进入例行职责

每天固定时间处理 exit tickets 的例子很典型：AI 不只是回答教师临时问题，而是在承担可重复、可预期、可复盘的工作。很多行业 cowork 产品也会走向这个方向，比如每周竞品追踪、每日销售线索整理、月度经营分析。

隐私和合规成为产品卖点

Anthropic 强调教师数据不用于训练、学生信息受 K-12 数据处理条款保护，并与 FERPA 对齐。越是进入行业工作流，cowork 产品越不能只强调效率，必须把数据边界、权限、合规和用户可控写进产品体验。

对 cowork 产品的启发

垂直 cowork 应该围绕角色设计：教师、投研、运营、销售、法务、研发，每个角色都有资料源、任务模板、输出标准和审核责任。产品越懂岗位，用户越少需要解释背景。

PM 简评：Claude for Teachers 是一个很好的垂直 cowork 样本。它说明行业产品不能只卖“通用 Agent 能力”，而要卖“这个岗位下周一就能用的工作流”。对公司组织用户来说，最有说服力的不是模型参数，而是它能否在既有制度和数据边界内稳定完成某类例行工作。

Google Gmail / Docs 更新：高频办公入口继续吸收轻量 cowork 能力

2026-07-14 至 2026-07-17 · Google Workspace Updates · Gmail Help me write custom refinement / Google Workspace Weekly Recap

Google Workspace 7 月 14 日发布 Gmail Help me write 自定义改写能力，用户可用自然语言要求 Gemini 修改邮件内容，而不再局限于 Polish、Formalize、Shorten 等固定选项；7 月 17 日周报还提到 Gemini in Docs 扩展多语言支持，并强调 Docs 里的 Gemini 可基于 Drive、Gmail、Chat 和 web 提供个性化上下文。

轻量入口会吃掉单点助手

邮件改写看似很小，但它发生在用户每天都用的 Gmail 里。用户不需要复制到第三方助手，也不需要学习复杂工作台。办公套件里的轻量 AI 会持续吞掉润色、改写、摘要、初步生成这些低门槛 cowork 需求。

自然语言成为功能控制层

从固定按钮到 prompt bar，说明办公软件正在把功能菜单抽象成意图输入。用户可以直接说“加一个截止日期”“语气更坚定”“补一段背景”，系统自动改写。这类交互会成为 AI 办公默认体验。

Workspace Intelligence 指向平台上下文

Docs 更新里提到 Gemini 可利用 Drive、Gmail、Chat 和 web 的上下文。这对独立 cowork 产品是压力：平台型办公软件天然掌握文件、邮件、会议和联系人，单点产品必须通过连接器或垂直场景弥补上下文劣势。

对 cowork 产品的启发

独立 cowork 不宜把核心价值押在单个写作动作上，而要做平台内置 AI 难以一次完成的跨系统任务：多源信息整合、持续追踪、审批协作、业务系统写回、复杂交付物生成。

PM 简评：Google 的动作提醒我们，cowork 的竞争不是只和其他 Agent 产品打，也和办公套件默认功能打。用户会优先使用离自己最近、摩擦最低的 AI。独立 cowork 要赢，必须提供比“在邮件里改写一句话”更深的闭环，例如跨邮件、文档、会议、表格和业务系统持续推进任务。
开源 / 自托管 Agent

OpenClaw v2026.7.1：Control UI、渠道、调度与成本视图集中升级

2026-07-13 · OpenClaw Docs · OpenClaw v2026.7.1 release notes

OpenClaw v2026.7.1 在 7 月 13 日发布，更新 Control UI、iOS/Android/macOS 应用、模型与 provider 支持、Codex 和连接式 coding-agent 工作流；Telegram、Slack、Discord、Apple Messages 等渠道也有更新，同时改进 Gateway、scheduled work、remote browser control、workspace terminals、sessions 和 goals。

自托管路线在补操作台

OpenClaw 的更新重点集中在 Control UI：会话标题、最近任务、pin/group/rename/fork/archive、多个 session 分屏、Tasks live view、成本和上下文展示。这些能力说明自托管 Agent 也在向“工作控制台”演进，而不是只提供后端运行能力。

多渠道是常驻 cowork 的入口策略

Telegram、Slack、Discord、Apple Messages 等渠道改进，意味着 Agent 的入口会越来越分散。用户不一定打开一个统一 app，而是在消息渠道、移动端、浏览器和本地控制台之间切换。cowork 产品需要让任务状态跨入口保持一致。

成本可见性是企业化基础

OpenClaw 把 provider、model、token、cache、daily spend、billing 等信息放进 UI，这对长期运行 Agent 很关键。只要 cowork 产品开始自动执行和定时运行，成本透明就从财务问题变成产品信任问题。

对 cowork 产品的启发

自托管或企业私有化 cowork 要同时解决四件事：多入口触达、会话管理、后台任务和成本治理。尤其在国内企业场景里，能否清楚显示任务、模型、费用、权限和日志，会直接影响采购信任。

PM 简评：OpenClaw 这条线的意义是提醒我们：除了大厂 SaaS，还有一类用户会选择可控、自托管、可接多 provider 的 Agent 基座。它的优势不在品牌，而在透明、可改、可审计。对 cowork 产品来说，未来可能不是纯云端或纯本地二选一，而是按数据敏感度、成本、入口和自动化程度做混合部署。

关键判断

1. Kimi K3 把国产 cowork 底座推到全球讨论中心，但产品胜负仍取决于能否把长上下文、视觉和推理能力落成稳定工作流。

2. 研发 Agent 仍是最成熟试验场：计划、后台任务、子 Agent、轮次限制、Canvas 标注、多仓库和 Slack 状态更新，正在形成可迁移到通用办公的产品范式。

3. 企业 cowork 的后台能力正在显性化：模板、身份、Memory Store、可观测、预算、事件流、审批和审计会成为采购前提。

4. 高频入口比独立首页更重要：Slack、Gmail、Docs、TRAE App、桌面 Work、消息渠道都在证明，Agent 要待在用户已经工作的地方。

5. 最终交付物越来越关键：网页自动发布、原生 PPTX、HTML 分享、Widgets/Dashboard 都说明 cowork 产品必须关心结果如何被查看、编辑、分享和上线。

6. 垂直化不是包装，而是角色化工作流：教育、研发、投研、销售、运营等场景都需要默认模板、专业资料源和合规边界。

7. 对国内 cowork 产品来说，本周的重点不是只追某一家竞品，而是同时观察国产模型能力、字节/阿里工具链、海外 Agent 入口和开源/自托管路线的合流。
